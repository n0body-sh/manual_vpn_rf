- Два сервера: промежуточный в Москве + выход за рубежом

- Нужен, если обычный VPN **уже не работает** или хотите подстраховаться заранее

```
Клиент ──HTTPS:443──▶ Bridge (Москва) ══▶ Exit (Амстердам) ──▶ Интернет
```

### Что такое белые списки и при чём тут chaining

С 2025–2026 года в РФ активно внедряют режим **белых списков** — когда доступны **только** одобренные ресурсы (госуслуги, банки, маркетплейсы). Весь остальной трафик блокируется, включая любые зарубежные IP.

Обычный VPN-сервер стоит за границей. При белых списках соединение к нему **не пройдёт** — ТСПУ заблокирует трафик на выходе из страны. Неважно, какой протокол — VLESS, WireGuard, OpenVPN — результат один.

**Chaining** повышает шансы: промежуточный сервер в России (bridge) принимает подключение как внутренний трафик, а затем пересылает всё на exit-ноду за рубежом.

> **Chaining — не гарантия обхода белых списков.** Результат зависит от вашего региона и провайдера. Схема работает, пока IP-подсети хостера bridge-ноды не отфильтрованы ТСПУ в вашем регионе:
>
> - **Москва, Санкт-Петербург** — chaining через VDS крупных хостеров (Timeweb, Selectel) работает стабильно. Подсети этих хостеров критичны для бизнес-инфраструктуры региона.
> - **FirstVDS (Москва)** — по состоянию на июнь 2026 замечены проблемы с доступностью московских IP. Если bridge на FirstVDS недоступен — попробуйте Timeweb или Selectel.
> - **Областные города** — возможны проблемы. Региональные ТСПУ могут фильтровать подсети хостеров агрессивнее.
> - **Пермь, Тверь** (и, вероятно, другие регионы) — замечены проблемы с доступностью VDS-подсетей. Chaining через VDS может не работать.
>
> Рекомендуем bridge-ноды у **крупных хостеров** и у **нескольких провайдеров** одновременно. Но **гарантий нет** — при жёстких белых списках в вашем регионе может потребоваться другой подход.

> **Дисклеймер:** автор не несёт ответственности за неработоспособность VPN в условиях белых списков. Это инструкция, а не гарантия — ситуация с блокировками постоянно меняется, и работающее сегодня решение может перестать работать завтра.

**Windows:**


- Ввести пароль из письма (символы при вводе не отображаются — это нормально)

**macOS / Linux:**
- Открыть терминал, набрать:
```bash
ssh root@<IP-адрес-сервера>
```

После подключения вы увидите командную строку сервера — можно начинать.

## 3. Настройка файрвола (защита сервера)

Файрвол — это фильтр, который блокирует все подключения к серверу, кроме разрешённых. Без него сервер будет открыт для всех.

### Почему не 443?

Порт 443 — первое, что сканируют ТСПУ и автоматические сканеры РКН. Если на вашем IP открыт 443 и за ним не настоящий веб-сайт, а VLESS Reality — это подозрительно: TLS-хендшейк с SNI `dl.google.com`, но IP не принадлежит Google. На случайном порте (например 47293) сканер его просто не найдёт — перебирать все 65535 портов на каждом IP нереально. Случайный порт — дополнительный слой защиты поверх криптографической маскировки Reality.

Сначала разрешаем нужные порты:

```bash
# Разрешаем SSH (удалённый доступ) — без этого потеряете доступ к серверу!
ufw allow ssh

# Порт для VPN — выбираем случайный из диапазона 10000-60000
# Например, 58431. Запомните его — он будет использоваться в подключении.
# НЕ используйте 443 — его сканируют в первую очередь!
ufw allow 58431/tcp
```

Теперь нужно выбрать порт для панели управления. Это должно быть **другое** случайное число от 10000 до 60000. Например, 39282. Запомните его.

```bash
# Замените 39282 на свой выбранный порт
ufw allow 39282/tcp
```

Включаем файрвол:

```bash
ufw enable
```

Система спросит подтверждение — набираем `y` и нажимаем Enter. Проверяем:

```bash
ufw status
```

Должно быть видно три разрешённых порта: 22 (SSH), порт VPN (58431) и порт панели (39282).

## 4. Установка панели управления (3X-UI)

3X-UI — это веб-панель для управления VPN-сервером. Через неё создаются подключения, добавляются пользователи, настраивается маршрутизация.

Запускаем установщик:

```bash
bash <(curl -Ls https://raw.githubusercontent.com/MHSanaei/3x-ui/master/install.sh)
```

Установщик задаст вопросы:

| Вопрос | Что отвечать |
|--------|-------------|
| Установить кастомные настройки? | `y` |
| Логин | Придумайте свой (НЕ используйте `admin`) |
| Пароль | Сложный, минимум 10 символов |
| Порт | Тот самый порт, который открыли в файрволе (например 39282) |

Запомните логин, пароль и порт — без них не зайти в панель.

### Проверяем что всё работает

Открываем браузер на своём компьютере (не на сервере!) и переходим по адресу:

```
http://<IP-сервера>:<порт>
```

Например: `http://85.123.45.67:39282`

Должна появиться страница входа. Вводим логин и пароль, которые задали при установке.

## 5. Защита панели

Прямо сейчас ваша панель доступна по прямому адресу. Боты и сканеры могут её найти. Чтобы этого не произошло — скрываем её за секретным путём.

В панели:

1. Заходим в **Настройки**
2. Находим **Путь панели (URL Root Path)**
3. Вписываем любой путь, например: `/mypanel/`
4. Сохраняем

Теперь панель доступна только по адресу:
```
http://<IP>:<порт>/mypanel/
```

Старый адрес без пути перестанет работать — это и есть защита.

## 6. Создание VPN-подключения (VLESS Reality)

VLESS Reality — самый современный протокол. Трафик маскируется под обычное HTTPS-соединение с крупным сайтом (Google). Блокировки его не видят.

В панели:

1. Заходим в **Подключения** (в левом меню)
2. Нажимаем **Добавить подключение**
3. Заполняем:

| Параметр | Значение | Комментарий |
|----------|---------|-------------|
| Протокол | `vless` | Не менять |
| Порт | `58431` | Тот самый порт, который открыли в файрволе |
| Transmission | `tcp` | |
| Security | `reality` | Обязательно reality, не TLS |
| Flow (Устанавливается в клиенте) | `xtls-rprx-vision` | Маскировка паттернов пакетов |
| uTLS | `chrome` | Маскировка под браузер Chrome |
| Dest (SNI) | `dl.google.com:443` | Сайт, под который маскируемся (443 — порт donor-сайта, не наш) |
| Server Names | `dl.google.com` | Тот же домен без порта |
| Private / Public Key | Нажать кнопку генерации | Сгенерируются автоматически |
| Short ID | Нажать кнопку генерации | Или оставить пустым |

4. Нажимаем **Добавить клиента** внизу формы
5. Вписываем имя (например, `phone` или `laptop`)
6. Нажимаем **Создать**

### Получаем ссылку для подключения

После создания в списке подключений появится новая строка. Нажимаем на иконку (QR-код или копирование) рядом с именем клиента — получаем ссылку вида:

```
vless://xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx@IP:58431?...
```

Эту ссылку нужно добавить в VPN-клиент на телефоне или компьютере (см. раздел 9).

> **Для варианта 2 (chaining):** сохраните эту ссылку отдельно — она понадобится при настройке bridge-ноды.

## 7. Установка Cloudflare WARP (обход детекта VPN)

### Зачем это нужно

Google, YouTube, Gemini и другие сервисы умеют определять, что ваш IP принадлежит дата-центру, а не домашнему провайдеру. В результате — ограничения, капчи, блокировки.

WARP — бесплатный сервис от Cloudflare. Он подменяет выходной IP сервера на «чистый» адрес из сети Cloudflare, который не числится в базах как датацентр.

### 7.1. Установка

Выполняем команды по порядку:

```bash
curl -fsSL https://pkg.cloudflareclient.com/pubkey.gpg \
  | gpg --dearmor -o /usr/share/keyrings/cloudflare-warp-archive-keyring.gpg

echo "deb [signed-by=/usr/share/keyrings/cloudflare-warp-archive-keyring.gpg] https://pkg.cloudflareclient.com/ $(lsb_release -cs) main" \
  > /etc/apt/sources.list.d/cloudflare-client.list

apt update && apt install -y cloudflare-warp
```

### 7.2. Настройка

```bash
# Регистрируем устройство в Cloudflare
warp-cli registration new

# Включаем режим прокси — WARP не трогает весь трафик, работает только как локальный прокси
warp-cli mode proxy
warp-cli proxy port 40000

# Подключаемся
warp-cli connect
```

### 7.3. Проверка

```bash
curl -x socks5://127.0.0.1:40000 https://www.cloudflare.com/cdn-cgi/trace
```

В выводе ищем две строки:
- `warp=on` — WARP активен
- `ip=104.28.xxx.xxx` — IP от Cloudflare (не ваш серверный)

Если видите `warp=on` — всё работает. Если нет — проверяем статус: `warp-cli status`.

### 7.4. Автозапуск

WARP автоматически стартует после перезагрузки сервера. Убедиться:

```bash
systemctl is-enabled warp-svc
```

Если ответ `enabled` — всё в порядке. Если нет:

```bash
systemctl enable warp-svc
```

## 11. Обход VPN для российских сервисов (настройка клиента)

### Зачем

Многие российские сервисы (Госуслуги, банки, Яндекс, VK, Ozon и др.) блокируют или ограничивают доступ с зарубежных IP. Если весь трафик идёт через VPN-сервер в Амстердаме — эти сервисы могут работать некорректно или вообще не пускать.

Решение — настроить VPN-клиент так, чтобы трафик к российским ресурсам шёл **напрямую через провайдера**, минуя VPN-туннель. Остальной трафик по-прежнему идёт через VPN.

```
Трафик к google.com  → через VPN-туннель → сервер → интернет
Трафик к gosuslugi.ru → напрямую, мимо VPN → интернет
```

> **Важно:** сервисы, для которых на сервере настроен WARP (Google, YouTube, Gemini и т.д.), должны по-прежнему идти **через VPN** — не добавляйте их в bypass.

### Hiddify (самый простой способ)

В Hiddify есть **встроенная функция маршрутизации по региону**. Достаточно выбрать Россию — и все российские сервисы автоматически пойдут напрямую, без VPN.

1. Открыть **Настройки** → **Маршрутизация**
2. **Регион** → выбрать **Россия (ru)**
3. Готово — Hiddify сам знает какие домены и IP относятся к РФ

Дополнительно можно включить **Блокировать рекламу** и **Обход LAN** в этом же разделе.

### v2rayTun (Android / iOS)

В v2rayTun есть готовые правила от сообщества:

1. Открыть **Правила трафика**
2. Включить функцию правил
3. Выбрать правило от сообщества: **".RU без VPN"**
4. Готово — весь трафик к российским сервисам пойдёт напрямую

### v2rayN (Windows)

1. Открыть v2rayN
2. **Настройки** → **Routing Setting** → **Advanced Function**
3. Выбрать пресет **«Все, кроме РФ»** — если доступен
4. Или добавить правила вручную:

```json
{
  "outboundTag": "direct",
  "domain": ["geosite:category-ru"]
}
```

```json
{
  "outboundTag": "direct",
  "ip": ["geoip:ru"]
}
```

5. Убедиться что правило `direct` стоит выше правил прокси для совпадающих доменов

### v2rayNG (Android)

1. Открыть v2rayNG
2. Нажать три точки → **Settings** → **Routing**
3. Выбрать **Bypass mainland** или **Custom**
4. В кастомных правилах добавить:
   - **Direct Domain**: `geosite:category-ru`
   - **Direct IP**: `geoip:ru`

### Ручной способ (если в приложении нет встроенной функции)

Правила выше (Hiddify → регион, v2rayTun → правила сообщества) покрывают большинство случаев. Ручная настройка нужна только если в вашем клиенте **нет** встроенной маршрутизации по региону.

В этом случае вручную добавьте домены в список прямых подключений:

```

domain:yandex.ru

domain:vk.com
domain:vk.ru

domain:avito.ru
domain:ria.ru

```

### Как проверить

1. Включить VPN
2. Открыть [whatismyip.com](https://whatismyip.com) → IP должен быть амстердамским (VPN работает)
3. Открыть [2ip.ru](https://2ip.ru) → IP должен быть **российским** (bypass работает)

Если оба условия выполнены — настройка верная: зарубежные сайты идут через VPN, российские — напрямую.

### Зачем это важно для безопасности

Раздельная маршрутизация — это не только удобство. С 2025–2026 года Минцифры [требует от российских IT-компаний](https://habr.com/ru/articles/1020080/) встраивать в свои приложения модули, помогающие обнаруживать VPN. Если российские приложения (Яндекс, Ozon, Госуслуги и др.) отправляют трафик **через VPN** — их серверы видят IP вашего VPN-сервера и могут сдать его РКН.

**Split routing (geosite:ru → direct)** решает эту проблему: российские приложения идут напрямую, мимо VPN — им нечего репортить.

Однако существует и более продвинутый вектор атаки: шпионский модуль может сканировать localhost-порты на устройстве и найти SOCKS5-прокси VPN-клиента, через который узнаёт exit IP напрямую. От этого split routing не спасает, но защищают другие меры из этого гайда:

| Мера | Как защищает |
|------|-------------|
| **Chaining (bridge + exit)** | Шпион узнает IP exit-ноды, но вы подключаетесь к bridge — другой IP. Забанят exit? Поднимете WARP или смените, bridge продолжит работать |
| **WARP на exit** | Шпион увидит IP Cloudflare, а не ваш сервер |
| **Split routing** | Российские приложения-шпионы не видят VPN-трафик |

> **Рекомендация:** не используйте клиент **Happ** — в нём обнаружена [критическая уязвимость](https://habr.com/ru/articles/1020080/), позволяющая дампить конфиги с ключами шифрования.

## 13. Решение проблем

### VPN подключается, но сайты не открываются
Проверить что Xray работает: `x-ui status`. Если нет — `x-ui restart`.

### Gemini/YouTube снова детектят VPN
WARP мог отключиться. Проверяем: `warp-cli status`. Если статус Disconnected — `warp-cli connect`, затем `x-ui restart`.

### Не заходит в панель
Проверить порт: `ufw status`. Если порта нет — `ufw allow <порт>/tcp`. Проверить панель: `x-ui status`.

### Забыли логин и пароль
```bash
x-ui settings    # показать текущие
x-ui reset       # сбросить на дефолт
```

### WARP не ставится
Убедиться что Ubuntu 22.04 или 24.04. На других версиях репозиторий Cloudflare может не работать.

### Встроенный WARP в 3X-UI (через кнопку WARP в панели) не работает
Это нормально — на многих VPS встроенный WireGuard-outbound падает с таймаутом. Рабочий вариант — `warp-cli` в режиме SOCKS5-прокси, как описано в этом гайде.

## Бонус. Маскировка сервера под обычный хостинг

### Зачем

Когда РКН сканирует IP-адреса, VPS без единого открытого стандартного порта — аномалия. Сервер с работающим сайтом на 443 — обычный хостинг, сканер проходит мимо.

| Без сайта | С сайтом на 443 |
|-----------|-----------------|
| Порт 443 закрыт, виден только случайный порт | Порт 443 → настоящий HTTPS-сайт |
| Подозрительно: VPS без сервисов | Легитимно: визитка, портфолио, блог |
| Проще попасть в чёрный список | Заблокировать = заблокировать «чей-то сайт» |

VLESS Reality при этом работает на своём случайном порту — никак не конфликтует с nginx.

### Шаг 1. Регистрация домена

Для полноценного HTTPS-сайта нужен домен. Подойдёт самый дешёвый — `.ru`, `.site`, `.online`.

Рекомендуем [Рег.ру](https://www.reg.ru/domain/new/?rlink=reflink-31613125) (скидка 5% по ссылке):

1. Переходим на [Рег.ру](https://www.reg.ru/domain/new/?rlink=reflink-31613125)
2. Ищем домен (например, `ivan-photo.ru` или `studio-design.site`)
3. Покупаем — от ~200₽/год для `.ru`

> Домен должен выглядеть правдоподобно — как настоящий сайт человека или маленькой компании. Не `myvpnserver.ru`.

### Шаг 2. Привязка домена к серверу (DNS)

В личном кабинете Рег.ру:

1. Заходим в **Мои домены** → выбираем домен
2. Переходим в **DNS-серверы / Управление зоной**
3. Добавляем **A-запись**:

| Поле | Значение |
|------|---------|
| Имя | `@` (корень домена) |
| Тип | `A` |
| Значение | IP вашего VPN-сервера |

4. Добавляем ещё одну A-запись для `www`:

| Поле | Значение |
|------|---------|
| Имя | `www` |
| Тип | `A` |
| Значение | IP вашего VPN-сервера |

5. Ждём 5-30 минут, пока DNS обновится

Проверяем:
```bash
ping ваш-домен.ru
```
Должен отвечать с IP вашего сервера.

> **Для chaining:** можно привязать один домен к exit-ноде, другой — к bridge. Или использовать поддомены: `exit.ваш-домен.ru` и `bridge.ваш-домен.ru`.

### Шаг 3. Установка nginx

На сервере:

```bash
apt install -y nginx
ufw allow 80/tcp
ufw allow 443/tcp
```

Проверяем: откройте `http://<IP-сервера>` в браузере — должна появиться стандартная страница nginx.

### Шаг 4. Скачиваем шаблон сайта

Нужна реалистичная HTML-страница. Бесплатные шаблоны:

| Источник | Что там | Ссылка |
|----------|---------|--------|
| HTML5 UP | Качественные адаптивные шаблоны | [html5up.net](https://html5up.net) |
| Start Bootstrap | Визитки, портфолио, лендинги | [startbootstrap.com](https://startbootstrap.com/themes/landing-pages) |
| Free CSS | 3000+ шаблонов | [free-css.com](https://www.free-css.com) |
| Cruip | Современные лендинги | [cruip.com/free-templates](https://cruip.com/free-templates/) |

Скачиваем любой шаблон и распаковываем:

```bash
# Пример: скачиваем шаблон и распаковываем в nginx
cd /tmp
wget https://html5up.net/identity/download -O template.zip
apt install -y unzip
unzip template.zip -d /var/www/html/
```

Проверяем: откройте `http://<IP-сервера>` — должен появиться красивый сайт.

> **Важно:** поменяйте текст в шаблоне на что-то своё (имя, описание, фото). Одинаковые шаблоны с дефолтным текстом на сотне IP — это тоже паттерн.

### Шаг 5. HTTPS-сертификат (Let's Encrypt)

```bash
apt install -y certbot python3-certbot-nginx
certbot --nginx -d ваш-домен.ru -d www.ваш-домен.ru
```

Certbot спросит email (для уведомлений об истечении сертификата) и согласие с условиями.

После установки сертификата сайт доступен по `https://ваш-домен.ru` с настоящим замком в браузере.

Автопродление уже настроено. Проверить:

```bash
certbot renew --dry-run
```

### Результат

```
Сканер РКН → стучится на IP:443 → получает HTTPS-сайт с валидным сертификатом → "обычный хостинг" → проходит мимо
Клиент VPN → подключается на IP:47293 → VLESS Reality → VPN работает
```

# Часть 2. Chaining (повышение устойчивости)

## 17. Файрвол bridge-ноды

## 19. Создание VPN-подключения на bridge (inbound)

Это «входная дверь» — сюда подключаются ваши устройства. Настройка аналогична exit-ноде, но с **другими ключами и другим donor-сайтом**.

В панели: **Подключения** → **Добавить подключение**

| Параметр | Значение | Комментарий |
|----------|---------|-------------|
| Протокол | `vless` | Не менять |
| Порт | `47293` | Тот самый порт, который открыли в файрволе |
| Transmission | `tcp` | |
| Security | `reality` | Обязательно |
| Flow (Устанавливается в клиенте) | `xtls-rprx-vision` | Маскировка паттернов — обязательно! |
| uTLS | `chrome` | |
| Dest (SNI) | `www.yandex.ru:443` | Маскируемся под Яндекс — российский donor для российского сервера (443 — порт donor-сайта, не наш) |
| Server Names | `www.yandex.ru` | |
| Private / Public Key | Нажать генерацию | **Новые** ключи, отличные от exit |
| Short ID | Нажать генерацию | |

> **Важно:** donor-сайт на bridge (`www.yandex.ru`) **должен отличаться** от donor-сайта на exit (`dl.google.com`). Bridge в России — donor тоже российский, чтобы TLS-хендшейк выглядел естественно для DPI.

Добавляем клиента (имя — `phone`, `laptop` и т.д.) → **Создать**.

## 20. Настройка пересылки на exit-ноду (outbound)

Здесь мы говорим bridge-ноде: **«весь входящий трафик пересылай на Амстердам»**.

### 20.1. Создаём исходящее подключение

В панели: **Настройки Xray** → **Исходящие подключения** → **+ Создать исходящее подключение**

Заполняем значениями из ссылки exit-ноды (которые записали в шаге 15):

| Поле | Значение | Откуда |
|------|---------|--------|
| Тег | `exit-node` | Придумываем сами |
| Протокол | `vless` | |
| Адрес | IP вашей exit-ноды | Из ссылки: после `@` |
| Порт | Порт exit-ноды | Из ссылки: после IP через `:` |
| UUID (ID) | UUID exit-ноды | Из ссылки: строка перед `@` |
| Encryption | `none` | **Обязательно!** Без этого Xray не запустится |
| Flow | `xtls-rprx-vision` | Из ссылки: параметр `flow=` |
| Безопасность | `reality` | |
| SNI | SNI exit-ноды | Из ссылки: параметр `sni=` |
| uTLS (Fingerprint) | `chrome` | Из ссылки: параметр `fp=` |
| Public Key | Public Key exit-ноды | Из ссылки: параметр `pbk=` |
| Short ID | **Только один** Short ID | Из ссылки: параметр `sid=` |

> **Частая ошибка:** поле **Encryption** оставляют пустым. Там **обязательно** `none` — иначе Xray выдаст ошибку при запуске.

> **Частая ошибка:** в **Short ID** вставляют все Short ID с exit-ноды. Нужен **один** — тот, что в ссылке.

Нажимаем **Создать исходящее подключение**.

### 20.2. Создаём правило маршрутизации

Направляем **весь трафик** через exit-ноду.

В панели: **Настройки Xray** → **Маршрутизация** → **+ Создать правило**

| Поле | Значение |
|------|---------|
| Inbound Tags | (пусто — все подключения) |
| Network | `tcp,udp` |
| Outbound Tag | `exit-node` |

**Создать правило** → **Сохранить** → **Перезапуск Xray** (красная кнопка).

> **Зачем это правило:** без него bridge отправит трафик в интернет напрямую через московский IP. Пользователь подключится, но увидит IP Москвы, а не Амстердама — весь смысл цепочки пропадёт.

## 21. Проверка chaining

### 21.1. Проверяем связь между серверами

На bridge-сервере (Москва):

```bash
curl -v --connect-timeout 5 https://<IP_EXIT>:<ПОРТ_EXIT> 2>&1 | head -20
```

Что должны увидеть:
- `Connected to <IP> port <PORT>` — связь есть
- `SSL certificate problem: self-signed certificate` — это нормально, Reality не отдаёт настоящий сертификат

Если `Connection timed out` — проверьте файрвол на exit-ноде (порт должен быть открыт).

```bash
ping -c 5 <IP_EXIT>
```

Ожидаемый пинг Москва → Амстердам: **30–50 мс**.

### 21.2. Подключаемся клиентом

1. В 3X-UI на **bridge** копируем ссылку подключения (`vless://...@<IP_МОСКВА>:47293?...`)
2. Добавляем в VPN-клиент (Hiddify / v2rayN / v2rayNG)
3. Подключаемся

### 21.3. Проверяем IP

- [whatismyip.com](https://whatismyip.com) → должен показать IP **exit-ноды** (Амстердам), **не** Москвы
- [2ip.ru](https://2ip.ru) → если настроен обход (раздел 11) → IP **вашего провайдера**

**Если whatismyip показывает IP exit-ноды — цепочка работает.**

## 23. Решение проблем (chaining)

### VPN подключается, но сайты не открываются

Проверить что Xray работает на bridge: `x-ui status`. Если не работает — `x-ui restart`. Логи: `x-ui log`.

### Клиент показывает IP Москвы, а не Амстердама

Маршрутизация на bridge настроена неправильно:
1. Outbound `exit-node` создан и активен?
2. Правило маршрутизации указывает на `exit-node`?
3. После изменений нажали **Перезапуск Xray**?

### Ошибка «encryption: none»

```
failed to build outbound: please add/set "encryption":"none"
```

В настройках outbound поле **Encryption** = `none`. Пустое поле — ошибка.

### Ошибка Short ID

Если в логах ошибки про Short ID — убедитесь, что в outbound указан **один** Short ID (из ссылки), а не несколько через запятую.

### Bridge перестал работать (IP забанили)

Bridge — это **расходник**. Если IP заблокировали:

1. Покупаете новый VDS в Москве на [FirstVDS](https://firstvds.ru/?from=1060968) (~200₽)
2. Повторяете шаги 16–20 (~15 минут)
3. Обновляете ссылку в VPN-клиенте
4. **Exit-нода не меняется** — она защищена за bridge'ом, её IP никто не видит

### Забыли логин/пароль bridge

```bash
x-ui settings    # показать текущие
x-ui reset       # сбросить на дефолт
```

## 24. FAQ

### Это законно?

Использование VPN в РФ не запрещено для физических лиц. Запрещено распространение средств обхода блокировок (ст. 15.8 ФЗ-149), но ответственность несут владельцы VPN-сервисов, а не пользователи.

### Bridge замедляет интернет?

Добавляется пинг Москва → Амстердам (~30-50 мс). Для веб-сёрфинга, YouTube и мессенджеров незаметно. Для онлайн-игр может быть ощутимо.

### Можно ли подключить несколько устройств?

Да. В 3X-UI создаёте нескольких клиентов — по одному на каждое устройство. Каждый получает свою ссылку.

### Нужен ли WARP на bridge?

Нет. WARP нужен только на exit-ноде — он подменяет выходной IP для Google и других сервисов. Bridge просто пересылает трафик.

### Можно ли иметь несколько bridge?

Да. Можно держать 2-3 bridge-ноды на разных провайдерах как резерв. Exit для всех одна.

# Часть 3. WebSocket-транспорт (обход ТСПУ на мобильном интернете)

Настройка WebSocket + TLS транспорта на bridge-ноде — альтернативный способ подключения, который работает на мобильном интернете и у проблемных провайдеров.

> **Предполагается**, что у вас уже настроена цепочка chaining (Часть 2): bridge в Москве + exit в Амстердам.

> **Проверено: июнь 2026.** Схема VLESS + WebSocket + TLS через bridge в Москве подтверждённо работает в **Москве** и **Санкт-Петербурге** — как на **домашнем интернете** (WiFi), так и на **мобильном** (МТС, Билайн, Мегафон). Трафик выглядит как обычный HTTPS к вашему домену — ТСПУ не может отличить его от визита на сайт.

> **Проблема белых списков и российских IP.** РКН включил в белые списки не все российские IP-адреса. Подсети VDS-хостеров (FirstVDS, Timeweb и др.) могут **не входить** в разрешённый список — а значит, даже подключение к bridge в Москве может блокироваться в вашем регионе. По состоянию на июнь 2026, у **FirstVDS замечены проблемы** с московскими адресами — рекомендуем пробовать Timeweb или Selectel в качестве альтернативы. WebSocket + TLS + домен повышает шансы на прохождение фильтров (трафик выглядит как обычный HTTPS к легитимному сайту), но **гарантий нет**. Если bridge недоступен — попробуйте VDS у другого хостера или в другом ДЦ.

## 25. Зачем нужен WebSocket-транспорт

VLESS Reality работает поверх TCP — и ТСПУ научился определять такие соединения. В 2026 году ряд провайдеров (особенно мобильные операторы: МТС, Билайн, Мегафон) шейпят или полностью блокируют TCP-соединения к VDS, которые выглядят «подозрительно» для DPI.

**WebSocket + TLS** — альтернативный транспорт, который решает эту проблему:

| Параметр | VLESS Reality (TCP) | VLESS + WS + TLS |
|----------|---------------------|-------------------|
| Транспорт | TCP | WebSocket поверх HTTPS |
| Что видит ТСПУ | TLS-хендшейк к чужому домену (dl.google.com, но IP — не Google) | Обычный HTTPS к **вашему** домену (IP совпадает) |
| Порт | Случайный (10000–60000) | 443 (стандартный HTTPS) |
| Мобильный интернет | Может шейпиться | Работает |
| WiFi | Работает | Работает |
| Нужен домен | Нет | Да (~100₽/год) |

```
Что видит ТСПУ:

Ваш телефон ──HTTPS:443──▶ bridge.ваш-домен.ru
                             ↑
                        Обычный TLS к обычному домену
                        Неотличимо от визита на сайт
```

> **WebSocket — дополнение, а не замена.** На bridge будут работать **оба** подключения параллельно:
> - VLESS Reality на случайном порту — для стабильных сетей (основной)
> - VLESS + WS + TLS на порту 443 — для мобильного интернета и проблемных провайдеров (запасной)

### Можно ли без домена?

**Да.** WebSocket работает и без домена — можно подключаться напрямую по IP bridge-сервера. В этом случае пропускаете разделы 26–28 (домен, DNS, сертификат) и создаёте inbound с `security: none` вместо `tls`.

Ссылка без домена:
```
vless://UUID@IP_BRIDGE:8443?encryption=none&security=none&type=ws&path=%2FMySecretPath123#WS-no-TLS
```

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| **WS без домена** | Быстро настроить, не нужен домен | Трафик не зашифрован TLS, провайдер видит содержимое WebSocket-фреймов |
| **WS + TLS + домен** | Полная маскировка под HTTPS, провайдер видит только «визит на сайт» | Нужен домен (~100₽/год) и сертификат |

> **Рекомендация:** если bridge-сервер доступен с вашего провайдера — начните с варианта **без домена** для проверки. Работает? Отлично. Потом можно добавить домен и TLS для надёжности. Главное — чтобы московский сервер был доступен с вашего провайдера.

## 27. Привязка домена к bridge (DNS)

Нужно создать DNS-запись, которая направит домен на IP вашего bridge-сервера.

В личном кабинете Рег.ру:

1. Заходим в **Мои домены** → выбираем домен
2. Переходим в **DNS-серверы / Управление зоной**
3. Добавляем **A-запись**:

| Поле | Значение |
|------|---------|
| Имя | `bridge` (или `vpn`, `connect` — любое слово) |
| Тип | `A` |
| Значение | IP вашего bridge-сервера (Москва) |

4. Сохраняем и ждём 5–30 минут

### Проверяем

На своём компьютере:

```bash
# macOS / Linux
dig bridge.ваш-домен.ru +short
```

Должен ответить с IP bridge-сервера. Если не резолвится — DNS ещё не обновился, подождите 15–30 минут.

> **Почему поддомен, а не корень?** Корневой домен (`ваш-домен.ru`) можно использовать для маскировки — повесить на него сайт (см. раздел «Бонус»). Поддомен `bridge.ваш-домен.ru` — отдельно для VPN-подключения.

## 28. SSL-сертификат на bridge

Для TLS нужен настоящий SSL-сертификат. Let's Encrypt выдаёт бесплатные сертификаты на 90 дней с автопродлением.

На bridge-сервере по SSH:

### 28.1. Установка certbot

```bash
apt install -y certbot
```

### 28.2. Открываем порт 80

Certbot использует порт 80 для подтверждения владения доменом:

```bash
ufw allow 80/tcp
```

### 28.3. Получаем сертификат

```bash
certbot certonly --standalone --non-interactive --agree-tos \
  --email ваш-email@example.com \
  -d bridge.ваш-домен.ru
```

> Замените `ваш-email@example.com` на реальный email — туда придёт уведомление если сертификат не продлится автоматически.

Если всё успешно, появится сообщение `Successfully received certificate` и файлы:

| Файл | Путь |
|------|------|
| Сертификат | `/etc/letsencrypt/live/bridge.ваш-домен.ru/fullchain.pem` |
| Приватный ключ | `/etc/letsencrypt/live/bridge.ваш-домен.ru/privkey.pem` |

### 28.4. Проверка автопродления

Certbot автоматически настраивает продление через systemd-таймер. Проверить:

```bash
certbot renew --dry-run
```

Если видите `Congratulations` — автопродление работает.

> **Важно:** порт 80 должен оставаться открытым в файрволе — certbot использует его для продления сертификата каждые 60–90 дней.

### 28.5. Привязка домена к панели 3X-UI

После получения сертификата привяжите домен к панели — тогда ссылки для клиентов будут автоматически генерироваться с доменом вместо IP.

В панели bridge: **Настройки панели**

| Параметр | Значение |
|----------|---------|
| Путь к сертификату (Public Key File Path) | `/etc/letsencrypt/live/bridge.ваш-домен.ru/fullchain.pem` |
| Путь к ключу (Private Key File Path) | `/etc/letsencrypt/live/bridge.ваш-домен.ru/privkey.pem` |
| Внешний домен (External IP / Domain) | `bridge.ваш-домен.ru` |

Нажмите **Сохранить** → **Перезапустить панель**.

После этого:
- Панель будет доступна по `https://bridge.ваш-домен.ru:<порт_панели>/<путь>/`
- Все новые ссылки клиентов будут содержать домен вместо IP

> **Частая ошибка:** если не указать сертификат в настройках панели, браузер покажет `NET::ERR_CERT_COMMON_NAME_INVALID` при попытке открыть панель по домену. Это происходит потому, что панель использует self-signed сертификат, выпущенный на IP, а вы обращаетесь по доменному имени. Решение — указать Let's Encrypt сертификат как показано выше.

## 29. Создание WebSocket-подключения

Теперь создаём новый inbound на bridge с WebSocket-транспортом.

### 29.1. Открываем порт 443

```bash
ufw allow 443/tcp
```

> TCP-порт 443 не конфликтует с VLESS Reality — тот работает на случайном порту. У вас будут оба подключения одновременно.

### 29.2. Создаём inbound в 3X-UI

В панели bridge: **Подключения** → **Добавить подключение**

**Вкладка «Основное»:**

| Параметр | Значение |
|----------|---------|
| Примечание | `ws-tls` (или любое понятное имя) |
| Протокол | `vless` |
| Порт | `443` |

**Вкладка «Поток» (Transmission):**

| Параметр | Значение |
|----------|---------|
| Transmission | `ws` |
| Path | Придумайте секретный путь (например `/MySecretPath123`) |

> Path — это секретный адрес внутри WebSocket-соединения. Придумайте длинную случайную строку — без неё к VPN не подключишься.

**Вкладка «Безопасность»:**

| Параметр | Значение |
|----------|---------|
| Security | `tls` |
| SNI | `bridge.ваш-домен.ru` |
| Публичный ключ (cert) | `/etc/letsencrypt/live/bridge.ваш-домен.ru/fullchain.pem` |
| Приватный ключ (key) | `/etc/letsencrypt/live/bridge.ваш-домен.ru/privkey.pem` |

> **Flow — оставить ПУСТЫМ!** Flow (`xtls-rprx-vision`) работает только с TCP/Reality. Для WebSocket его указывать **нельзя** — подключение не будет работать.

Нажимаем **Добавить клиента** → вписываем имя → **Создать**.

### 29.3. Маршрутизация

Если у вас уже настроена маршрутизация из Части 2 (правило `tcp,udp → exit-node`), то WebSocket-трафик **автоматически** пойдёт через exit-ноду в Амстердам. Дополнительных правил создавать не нужно.

> Если правило маршрутизации привязано к конкретному inbound тегу — добавьте тег нового WS-inbound в список входящих тегов правила.

## 32. Решение проблем (WebSocket)

### Таймаут при подключении через WS+TLS

Проверяем по порядку:

1. **DNS:** `ping bridge.ваш-домен.ru` → должен отвечать с IP bridge
2. **Порт:** на bridge по SSH: `ufw status | grep 443` → должен быть `443/tcp ALLOW`
3. **Сертификат:** `certbot certificates` → статус `VALID`
4. **Xray:** `x-ui status` → `running`
5. **Xray слушает:** `ss -tlnp | grep :443` → должен показать `xray`

### Сертификат не получается (certbot ошибка)

- Порт 80 должен быть открыт: `ufw allow 80/tcp`
- Порт 80 не должен быть занят nginx или другим сервисом: `ss -tlnp | grep :80`
- DNS должен указывать на IP bridge: `dig bridge.ваш-домен.ru +short`
- Если на сервере стоит nginx — используйте `certbot --nginx` вместо `--standalone`

### WS+TLS подключается, но IP не амстердамский (показывает Москву)

Маршрутизация не перенаправляет WS-трафик на exit-ноду:

1. Проверьте правило маршрутизации: Network = `tcp,udp`, Outbound Tag = `exit-node`
2. Если правило привязано к конкретному inbound тегу — добавьте WS-inbound
3. **Обязательно** нажмите **Перезапуск Xray** после изменений

### Ошибка «Flow is not supported» или «invalid config»

В inbound WS+TLS поле **Flow должно быть пустым**. Flow (`xtls-rprx-vision`) работает только с TCP/Reality — для WebSocket его указывать нельзя.

### Сертификат истёк (перестал подключаться через 90 дней)

```bash
certbot renew
x-ui restart
```

Проверить автопродление: `systemctl list-timers | grep certbot` — должен быть активный таймер.

### IPsec/L2TP мешает (QUIC-пакеты дропаются)

Если на bridge-сервере настроен IPsec VPN (L2TP/IPsec), правило `DROP ctstate INVALID` в iptables может блокировать UDP-трафик. Для WebSocket (TCP) это не проблема, но если будете добавлять другие UDP-протоколы — проверьте:

```bash
iptables -L INPUT -n --line-numbers | head -10
```

Если видите `DROP ctstate INVALID` **перед** UFW — добавьте правило-исключение:

```bash
iptables -I INPUT 2 -p udp --dport <ваш_порт> -j ACCEPT
```
